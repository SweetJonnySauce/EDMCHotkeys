# Wayland Side-Specific Modifiers via keyd Integration Plan

Status: Draft  
Owner: EDMCHotkeys  
Last Updated: 2026-03-07

## Objective
Provide side-specific modifier support on Linux Wayland with the lowest user configuration burden by integrating with `keyd` as an external dependency, while keeping EDMCHotkeys itself unprivileged.

## Strategy Summary
- Do not build a privileged in-plugin input backend.
- Use `keyd` for side-specific key capture.
- Keep EDMCHotkeys as an authenticated action receiver (binding-id activation path).
- Automate keyd config export from EDMCHotkeys bindings.
- Prompt users with the required reload command now; add an opt-in helper script later.

## Decisions Captured
- Backend mode decision:
  - New backend mode name: `wayland_keyd`.
  - Backend implementation module: `edmc_hotkeys/backends/wayland_keyd.py`.
  - Backend class name: `WaylandKeydBackend`.
  - Backend runtime name/identifier: `linux-wayland-keyd`.
  - `auto` on Wayland prefers `wayland_keyd` when keyd is detected as healthy.
  - Health criterion for selection:
    - primary: keyd service active via `systemctl is-active keyd`
    - fallback: keyd daemon process detected via `pgrep -x keyd` on non-systemd environments
  - No explicit integration enable flag is required for keyd path in normal `auto` operation.
  - Existing Wayland backends are slated for deprecation after keyd rollout stabilizes.
  - Deprecation is documented without a committed removal milestone in this plan.
- Artifact naming decision:
  - New preferred keyd artifact uses `linux-wayland`.
  - Current portal artifact is renamed to `linux-wayland-portal`.
  - Current GNOME bridge artifact is renamed to `linux-wayland-gnome-bridge`.
  - Naming migration is implemented in build/release logic only for this plan.
- Export location decision:
  - Generated keyd mapping artifacts live under plugin-local `keyd/runtime/` and are the authoritative source generated by EDMCHotkeys.
  - Canonical generated mapping filename: `keyd/runtime/keyd.generated.conf`.
  - Export state metadata filename: `keyd/runtime/export_state.json`.
  - `keyd/runtime/` is created at runtime (not shipped in release artifacts).
  - EDMCHotkeys runtime writes only plugin-local files; it does not directly mutate system keyd config paths.
- Apply model decision:
  - Applying generated mappings to active keyd config remains a separate apply/reload step (manual command prompt now, opt-in helper later).
  - Standard active keyd target path: `/etc/keyd/edmchotkeys.conf`.
  - Runtime logs/prompts must render the source path from resolved `plugin_dir` (do not hardcode path text at runtime).
  - Default suggested command (systemd):
    - `sudo install -m 0644 /home/jon/edmc_plugins/EDMCHotkeys/keyd/runtime/keyd.generated.conf /etc/keyd/edmchotkeys.conf && sudo systemctl restart keyd`
  - Fallback suggested command (non-systemd):
    - `sudo install -m 0644 /home/jon/edmc_plugins/EDMCHotkeys/keyd/runtime/keyd.generated.conf /etc/keyd/edmchotkeys.conf` and restart keyd using the host init system.
  - When no init-system-specific command can be determined, log this generic one-liner: `Restart keyd manually using your init system (for example: sudo service keyd restart).`
  - No per-distro command override config in this rollout; revisit only if support data shows need.
- Tradeoff acceptance:
  - Accepted pros: self-contained plugin artifacts, no root writes from plugin runtime, easier upgrade-safe generation.
  - Accepted cons: possible source-vs-active drift and need for explicit apply/reload workflow.
- Export policy decision:
  - Export active profile only (avoid cross-profile hotkey collisions in generated keyd mappings).
  - Export writes/reload prompts are skipped when `bindings_hash` is unchanged.
  - Invalid bindings: skip with warning in EDMC logs.
  - Conflicts: first binding wins, later conflicting rows are skipped with warning in EDMC logs.
  - Warning message templates are standardized:
    - invalid/unsupported: `keyd export skipped invalid binding: profile=<profile> action=<action_id> chord=<chord> reason=<reason>`
    - conflict first-wins: `keyd export conflict (first-wins): profile=<profile> chord=<chord> kept=<kept_action_id> skipped=<skipped_action_id>`
  - Per-device keyd sections are out of scope for this rollout and deferred to a later phase.
- Observability decision:
  - Integration status/reporting is logs-only (no settings UI status panel in this phase).
  - On startup, emit an info log confirming selected backend (`wayland_keyd` or fallback backend) when mode is `auto`.
- Dry-run decision:
  - Dry-run/export-preview is provided on the existing keyd export script via `--dry-run`.
  - Dry-run scope is export preview only (render would-be `keyd.generated.conf` output without writing/applying).
  - Dry-run output mode for this rollout: stdout only.
- Export-state schema decision:
  - `keyd/runtime/export_state.json` is required and versioned with semantic schema version.
  - `state_schema_version` lifecycle is controlled by EDMCHotkeys release process.
  - Initial proposed schema (v`1.0.0`) fields:
    - `state_schema_version` (string, semver; initially `1.0.0`)
    - `active_profile` (string)
    - `bindings_hash` (string)
    - `generated_at_utc` (string, ISO 8601)
    - `generated_path` (string; default `keyd/runtime/keyd.generated.conf`)
    - `apply_target_path` (string; default `/etc/keyd/edmchotkeys.conf`)
    - `reload_required` (bool)
    - `last_reload_prompt_command` (string)
  - Unknown/new fields must be tolerated.
  - Incompatible major schema versions must trigger warning logs, rebuild export state from current bindings, and continue.
- Env-var management decision:
  - Do not add new long-term env-var surface for keyd integration when equivalent config-file keys exist.
  - Existing env-var overrides are formally deprecated once config-file rollout is complete.
  - Deprecation behavior must include explicit warning logs; eventual removal timing remains a release-policy decision.
  - No additional warning-frequency policy is introduced in this plan.
- Wayland backend deprecation messaging decision:
  - Existing Wayland backend modes are formally deprecated in project planning/release policy.
  - Do not emit runtime deprecation warnings for current non-keyd Wayland backends in this rollout.

## Open Questions
- None currently blocking implementation planning.

## Scope
- Add a Wayland + keyd integration mode adjacent to existing backend modes.
- Auto-export keyd mappings from `bindings.json` on startup and settings save.
- Add release artifact support for a dedicated keyd-oriented variant.
- Add scripts/tests for install, verify, rollback, and troubleshooting behavior.
- Add config-file based runtime settings to reduce env-var burden.
- Add dry-run/export-preview tooling for keyd mapping generation.
- Add semantic versioning and compatibility handling for `export_state.json`.

## Non-Goals
- Do not bundle or redistribute `keyd`.
- Do not remove existing backends (`wayland_portal`, `wayland_gnome_bridge`, `x11`, `windows`) in the initial keyd rollout.
- Do not remove environment variable overrides.
- Do not intermix runtime logic/state/config between this keyd path and `wayland_gnome_bridge`.
- Do not include documentation/release-note authoring updates in this implementation plan.

## Backend Isolation Requirement
- The keyd integration path must remain an independent backend mode with isolated wiring.
- Do not reuse `wayland_gnome_bridge` sender-sync lifecycle or companion-extension flows for keyd mode.
- Do not share mutable runtime state between keyd mode and `wayland_gnome_bridge`.
- Use explicit mode-scoped config keys and diagnostics so behavior remains attributable to one backend at a time.
- Backend selector behavior must ensure only one backend is active; no hybrid execution between keyd and GNOME bridge paths.

## Backend Selection Requirement
- Introduce new backend mode `wayland_keyd` that sits alongside existing backend modes.
- On Wayland, `auto` mode must prefer the keyd integration backend when keyd is detected and healthy.
- If keyd is not detected/healthy, `auto` must fall back to existing Wayland selection behavior with explicit diagnostics.
- Explicit mode selection must continue to override `auto`.
- Keyd detection for selector decisions must use the same primary/fallback checks as the apply model (`systemctl` first, then process detection).
- When mode is `auto`, startup logs must include a single info line indicating the selected backend and selection reason.

## Configuration Requirement
- Runtime/backend options currently configured via environment variables must also be configurable via plugin-local INI files.
- Add repo-owned `config_template.ini` containing all supported backend/runtime keys with defaults and comments.
- Release build must generate and ship artifact-specific `config.defaults.ini` from the template, including only keys required for that artifact/backend path.
- User-local overrides must use `config.ini` (separate from `config.defaults.ini`).
- Plugin startup must resolve configuration with explicit precedence:
  - `env > config.ini > EDMC config > config.defaults.ini > code defaults`
- Effective config source must be logged at startup for diagnostics.
- Release packaging must never overwrite user-edited `config.ini`.
- `config.ini` is auto-created from `config.defaults.ini` when missing.

## Binding Export Requirement
- EDMCHotkeys must export active bindings to keyd configuration in a deterministic, idempotent way.
- Export must run automatically on plugin startup and after settings changes affecting bindings.
- Export output must be mode-scoped for this keyd integration path and must not reuse `wayland_gnome_bridge` export/sync flows.
- Export output path must be plugin-local `keyd/runtime/` and treated as the authoritative generated source.
- Export scope is active profile only.
- Export must skip rewriting output and skip reload prompt when computed `bindings_hash` is unchanged.
- Apply/reload docs/scripts must target `/etc/keyd/edmchotkeys.conf` as the default active keyd path for this rollout.
- Export must define behavior for unsupported/invalid/conflicting bindings:
  - invalid/unsupported rows: skip with explicit warnings in EDMC logs
  - conflicts: first wins, later rows skipped with explicit warnings in EDMC logs
  - warning messages should follow the standardized templates from Decisions Captured
  - do not abort entire export unless output cannot be written
- Export changes requiring keyd reload must trigger a clear manual command prompt in logs now.
- Prompting must emit an exact command path for systemd hosts and explicit fallback guidance for non-systemd hosts, using resolved `plugin_dir` in runtime logs.
- If a specific non-systemd restart command cannot be detected, emit the generic one-liner restart guidance defined in Decisions Captured.
- Later opt-in helper automation must preserve a safe fallback to manual command prompting.
- Provide a dry-run/export-preview command to render would-be output without writing files.
